# Dart Binding Generation Roadmap

This roadmap moves the Dart integration away from manually maintaining every
low-level AngularTS binding. The target state is a generated raw JS facade layer
plus a small handwritten ergonomic Dart API.

## Goal

- Generate repetitive Dart JS interop facades from the public AngularTS API.
- Keep curated Dart APIs handwritten where they improve safety and usability.
- Make CI fail when generated bindings or parity metadata are stale.
- Preserve the existing strict Dart analyzer and browser smoke test workflow.

## Source Of Truth

Primary input:

```text
@types/namespace.d.ts
```

Supporting inputs:

```text
integrations/dart/tool/generator-overrides.json
integrations/dart/NG_NAMESPACE_PARITY.md
```

Closure externs remain generated by the Closure integration. Dart should not
depend on Closure extern shape unless a specific API is easier to recover from
extern JSDoc than from TypeScript declarations.

## Target Layout

```text
integrations/dart/
  lib/src/generated/ng_facades.dart
  tool/generate_dart_bindings.mjs
  tool/check_generated_bindings.mjs
  tool/generator-overrides.json
```

Generated code should contain the low-level JavaScript boundary. Handwritten
files should contain ergonomic Dart types, builders, tokens, and helpers.

## Phase 1: Generator Skeleton

Create:

```text
integrations/dart/tool/generate_dart_bindings.mjs
integrations/dart/tool/generator-overrides.json
integrations/dart/lib/src/generated/ng_facades.dart
```

Generator requirements:

- Parse `@types/namespace.d.ts` with the TypeScript compiler API.
- Find `export namespace ng`.
- Extract public type declarations.
- Extract interface/type members:
  - properties
  - methods
  - call signatures
  - optional parameters
- Generate a deterministic Dart file.

Initial type mapping:

| TypeScript | Dart |
| --- | --- |
| `string` | `String` |
| `boolean` | `bool` |
| `number` | `num` |
| `void` | `void` |
| `Promise<T>` | `JSPromise<JSAny?>` |
| arrays | `JSArray<JSAny?>` or `Object?` |
| broad unions | `Object?` |
| unknown/dynamic | `Object?` |
| function types | `JSFunction` |
| DOM types | `package:web` types via overrides |

Acceptance criteria:

```sh
node integrations/dart/tool/generate_dart_bindings.mjs
dart format integrations/dart/lib/src/generated/ng_facades.dart
make -C integrations/dart analyze
```

Status: implemented.

## Phase 2: Freshness Check

Add a generated-output check that fails when committed generated bindings are
stale.

Create:

```text
integrations/dart/tool/check_generated_bindings.mjs
```

Required behavior:

- Run the generator into a temporary file or memory buffer.
- Compare against `lib/src/generated/ng_facades.dart`.
- Print a concise diff summary or regeneration command on failure.

Add Makefile targets:

```make
generate:
	node tool/generate_dart_bindings.mjs
	dart format lib/src/generated/ng_facades.dart

generate-check:
	node tool/check_generated_bindings.mjs
```

Update `check`:

```make
check: generate-check analyze test parity runtime-test
```

Acceptance criteria:

```sh
make -C integrations/dart generate
make -C integrations/dart generate-check
make -C integrations/dart check
```

Status: implemented.

## Phase 3: Member-Level Parity

The current parity check only verifies that every public `ng` type has an entry
in `NG_NAMESPACE_PARITY.md`. Add member-level coverage.

Required behavior:

- Compare public members in `@types/namespace.d.ts`.
- Verify each member is generated, manually implemented, or explicitly ignored.
- Report missing members with type and member name.

Override metadata uses concrete member names only:

```json
{
  "renames": {
    "StateService.is": "isState"
  },
  "manual": [
    "SomeType.someErgonomicMember"
  ],
  "unsupported": [
    "SomeType.unsupportedMember"
  ]
}
```

Acceptance criteria:

```sh
make -C integrations/dart parity
```

must fail for a missing public member, not just a missing public type.

Status: implemented through `tool/check_ng_namespace_members.mjs`.

## Phase 4: Replace Empty Facades First

Start replacing manual low-value facades with generated ones.

Good first targets:

- empty service facades
- simple provider `$get` members
- property-only records
- methods with `Object?` input/output

Do not replace yet:

- `NgModule`
- `Token<T>` and DI helpers
- `StateDeclaration`
- `RequestConfig`
- `AppComponent`
- any builder that converts Dart objects into AngularTS runtime shape

Acceptance criteria for each migrated group:

```sh
make -C integrations/dart generate
dart format integrations/dart/lib integrations/dart/test
make -C integrations/dart analyze
make -C integrations/dart test
make -C integrations/dart parity
make -C integrations/dart check
git restore dist
```

Status: implemented. Constructor-only service wrappers and low-risk provider
wrappers now inherit generated raw facades where a generated counterpart exists.
The generator also emits constructor-only bases for public object types with no
generated members, so empty facades and facades with fully manual ergonomic
members can still share the generated base layer. Curated typed members are
explicitly marked manual in generator overrides. Simple service facades such as
`AriaService` now inherit generated methods directly when the generated
signature already matches the public TypeScript surface. Callable-property
service APIs can be opted into method generation, and rest parameters are
expanded to a bounded optional argument list; `LogService` now uses this path
instead of handwritten logging methods. Required primitive parameters are
inferred for generated methods, and generated methods can carry explicit Dart
type parameters plus facade-return wrapping when TypeScript has a generic JS
object return such as `WebComponentService.createElementScope<TState>()`.
This lets small facades such as
`SceDelegateService` inherit generated methods without losing string argument
typing. Non-null primitive return types are inferred from TypeScript too, so
facades such as `SceService` can move to generated methods without weakening
boolean return contracts. Callable provider properties such as
`SceDelegateProvider` can now use generated method facades as well. Callable
parameters are inferred as `JSFunction`, and targeted integer return overrides
cover count APIs so `PubSubService` and `TopicService` can inherit generated
methods while keeping their previous public Dart shape. The same callable
parameter inference lets `TransitionService` expose generated hook registration
methods with callback parameters typed as `JSFunction`. Explicit renames can
preserve valid Dart API names that would otherwise be conservatively avoided,
such as `StateRegistryService.get`. `tool/check_generated_base_usage.mjs`
enforces that handwritten facades extend generated bases when a generated base
exists. Callable-interface methods such as `InterpolateService.startSymbol()`
and `InterpolateService.endSymbol()` now come from the generated base instead
of handwritten raw calls. Optional callable properties are unwrapped when
collecting call signatures, so members such as `TranscludeFn.isSlotFilled()`
can also be generated instead of handwritten. Direct string-only attribute
helpers such as `Attributes.$normalize()`, `$addClass()`, `$removeClass()`, and
`$updateClass()` now come from generated coverage. `$observe()` has a generated
raw member and a handwritten public adapter because it turns a Dart callback
into a JavaScript callback and returns a Dart disposer. Thin object-forwarding
helpers such as `WasmService.createScopeAbi()` also come from generated
coverage.
Generated parameter-type overrides can preserve a stricter Dart input type
without forcing a handwritten method; `WasmService.scope()` now takes
`Scope<Object?>` from generated coverage and `dartToJs()` unwraps objects that
implement `JsConvertible`. Callable facades now infer the same argument and
promise return types as normal methods, so `WasmService.call()` is generated
with a required `String` source and `JSPromise<JSAny?>` result. Writable scalar
properties now get generated setters too, removing handwritten setter-only
wrappers from provider and location facades. Callback-to-promise methods such
as `AnimateService.transition()` are generated with stricter `JSFunction` and
`JSPromise<JSAny?>` types. Simple optional-argument methods such as
`AnimateService.cancel()` also come from generated coverage when no Dart
adapter is needed. `InjectableFactory` implements `JsConvertible`, so simple
provider registration methods such as `AnimateProvider.register()` can use the
generated raw method without losing annotated-array conversion. Dart value
objects such as `CookieOptions` can also implement `JsConvertible`, allowing
`CookieService.put()`, `putObject()`, and `remove()` to come from generated
coverage while preserving option-object conversion. HTTP request config objects
use the same hook, so `$http.get()`, `delete()`, `head()`, `post()`, `put()`,
and `patch()` are generated with `String` URL parameters and
`JSPromise<JSAny?>` results while still accepting Dart config objects.
Animation option objects also convert themselves at generated boundaries, so
all `AnimateService` methods now come from generated coverage without losing
preset or native-options conversion.
`StateDeclaration` now uses the same conversion hook, allowing non-chainable
`StateService` helpers such as `reload()`, `go()`, `target()`,
`transitionTo()`, `isState()`, `includes()`, `href()`, and `get()` to come from
generated coverage while keeping state-declaration arguments usable.
Generated methods that return their own or an assignable TypeScript facade type
now return `this`, which lets simple fluent wrappers such as
`FilterProvider.register()` and `$provide` registration methods move out of
handwritten code. Required parameter inference now merges overloads before
falling back to `Object?`, so both `StateService.lazy()` and the overloaded
`StateService.state()` come from generated coverage while preserving the
`StateDeclaration`-or-`String` first argument.
`NgModule` now inherits the generated `name` getter and keeps only the typed
registration methods marked manual, so parity metadata no longer hides the whole
module behind a wildcard.
`StateDeclaration` also has generated raw facade member coverage; the
handwritten Dart value object remains as the public ergonomic constructor and
conversion layer.
Manual `NgModule` registration methods now route injectable factories, option
maps, and value-object options through the same `unsafe.dartToJs()` boundary
used by generated methods.
Web component value objects also convert themselves at generated boundaries, so
`WebComponentService.defineAppComponent()` and
`WebComponentService.defineElement()` are generated with
`CustomElementConstructor` return overrides. The generator can also attach a Dart
method type parameter and wrap raw facade returns, so
`WebComponentService.createElementScope<TState>()` is generated while still
returning `Scope<TState>`.

## Phase 5: Typed Overrides

Improve generated types incrementally using overrides.

Override categories:

- DOM type mapping to `package:web`.
- reserved word renames.
- forced nullable/non-nullable return types.
- generic erasure rules.
- callable-interface handling.
- manual ergonomic type substitutions.

Examples:

```json
{
  "types": {
    "HTMLElement": "HTMLElement",
    "Element": "Element",
    "AbortSignal": "AbortSignal"
  },
  "returnTypes": {
    "SceService.isEnabled": "bool",
    "LocationService.getPath": "String"
  },
  "parameterTypes": {
    "Scope": "Scope<Object?>"
  }
}
```

Acceptance criteria:

- Generated code stays analyzer-clean.
- Overrides are documented by purpose.
- No override exists only to silence the generator without a reason.

Status: started. The generator infers nullable `String`, `bool`, `num`, and
`JSPromise<JSAny?>` return types for simple TypeScript members, still supports
explicit member return type overrides, supports narrow TypeScript type-name to
Dart parameter-type overrides, maps configured platform types to `package:web`,
and rejects stale `returnTypes`, `parameterTypes`, and `types` entries in member
parity checks. Explicit manual overrides are checked by
`tool/check_manual_overrides.mjs`, so override metadata cannot hide a missing
handwritten Dart member.

## Phase 6: Handwritten Ergonomic Layer

Keep and document the high-level Dart layer.

Handwritten APIs should:

- expose Dart-native constructors and value objects;
- convert to JS only at runtime boundaries;
- avoid exposing generated internals unless needed;
- delegate raw member access to generated facades where practical.

Generated APIs should:

- stay deterministic;
- stay thin;
- avoid domain-specific behavior;
- avoid complex Dart object conversions.

Acceptance criteria:

- `README.md` explains generated vs handwritten layers.
- `COVERAGE.md` distinguishes raw generated coverage from ergonomic coverage.

Status: implemented.

## Phase 7: CI Contract

The final check contract should be:

```sh
make -C integrations/dart check
```

This must verify:

- generated Dart bindings are fresh;
- Dart formatting is stable;
- analyzer is clean;
- browser tests pass;
- type parity is complete;
- member parity is complete;
- override metadata is sorted, current, and backed by generated or handwritten
  Dart code;
- the Dart demo compiles against current `dist`;
- Playwright smoke test passes.

Status: implemented through the `generate-check`, `format-check`, `analyze`,
`test`, `parity`, `generated-base-check`, and `runtime-test` targets. The
`parity` target includes type parity, member parity, and manual override
integrity checks. Current member parity is fully generated, with no manual or
unsupported member overrides.

## Done Definition

The migration is complete when:

- Adding a new public `ng` namespace member updates Dart by running the generator
  or adding an explicit override.
- Manual repetitive facade work is no longer required.
- Missing generated bindings fail CI.
- Handwritten Dart files contain only ergonomic APIs, conversion logic, and
  intentionally curated wrappers.
