# Messages Directive Internals

This directory owns AngularTS `ngMessages` error/message rendering. The
implementation in `messages.ts` is centered on a parent `NgMessageCtrl` that
watches a message collection and coordinates transcluded `ngMessage`,
`ngMessageExp`, `ngMessageDefault`, and `ngMessagesInclude` children.

## Responsibilities

- Watch an error/message collection and display matching messages.
- Preserve DOM order when choosing the first matching message.
- Support single-message mode and `multiple` / `ng-messages-multiple` mode.
- Support static message keys, dynamic message expressions, default messages,
  and included remote message templates.
- Attach and detach transcluded message elements with animation support.
- Deregister messages when structural directives remove message nodes or their
  owning scopes.

## Public Surface

- `ngMessagesDirective($injector, $parse, $attributes)`: builds the root
  `ngMessages` directive and controller.
- `ngMessagesIncludeDirective($templateRequest, $compile, $attributes)`: loads
  and compiles external message templates into an existing `ngMessages`
  container.
- `ngMessageDirective`: registers a static `ng-message` / `when` message.
- `ngMessageExpDirective`: registers a dynamic `ng-message-exp` / `when-exp`
  message.
- `ngMessageDefaultDirective`: registers the fallback default message.

Public behavior exposed through directives includes `ng-messages`, `for`,
`multiple`, `ng-messages-multiple`, `ng-messages-include`, `src`,
`ng-message`, `when`, `ng-message-exp`, `when-exp`, and
`ng-message-default`.

## Core Model

`NgMessageCtrl` owns the current collection, registered message instances, an
optional default message, and the container active/inactive state. Child
messages are represented by `MessageInstance` objects with `test()`,
`attach()`, and `detach()` methods. Non-default messages are keyed through the
comment node generated by element transclusion.

The main flow is:

1. `ngMessagesDirective` creates `NgMessageCtrl` and watches the configured
   collection expression.
2. Each `ngMessage` child registers a `MessageInstance` against its transclude
   comment anchor.
3. `_render()` sorts registered messages by document position and tests them
   against truthy collection keys.
4. Matching messages are attached; unmatched messages are detached.
5. The default message is attached only when there are truthy collection keys
   and no concrete message matched.
6. The container toggles `ng-active` and `ng-inactive`.

Important invariants:

- Message priority follows rendered DOM order, not registration order.
- In single-message mode, only the first matching message is attached.
- In multiple-message mode, each truthy collection key can match at most one
  message.
- Default messages must not render when another message matched.
- Detached or externally destroyed message nodes must deregister from the
  parent controller.

## Lifecycle

The root directive creates a controller and starts watching the message
collection. Message directives use element transclusion, so the original node is
replaced by a comment anchor. Registration happens during link, but actual DOM
insertion happens only when the controller decides the message matches.

When a message attaches, the transcluded element is inserted after the comment
anchor and receives an attach id. If that element is destroyed by another
structural directive, the `$destroy` listener checks the attach id, deregisters
the message, detaches the current element, and destroys the transcluded scope.
If the containing scope is destroyed before a message ever attaches, the scope
destroy listener still deregisters it from the controller.

## Scheduling And Ordering

- Collection changes render through the scope watch installed by
  `NgMessageCtrl`.
- Message registration and deregistration call `reRender()`, which batches the
  follow-up render into a resolved-promise microtask.
- Included templates call `reRender()` after their downloaded content is
  compiled and linked.
- Message matching is synchronous inside `_render()`.
- Animation-aware attach/detach uses `enter`, `leave`, or `setClass` when an
  animation service is available for the affected node.

## Data Structures

- `_messages`: maps generated message keys to `{ message, comment }` records.
- `_default`: stores the optional default `MessageInstance`.
- `_cachedCollection`: stores the last collection used for deferred re-renders.
- `_renderLater`: prevents duplicate microtask renders.
- `_latestKey`: generates stable registration keys for comment anchors.
- `_nextAttachId`: generates ids that distinguish current attachments from
  stale destroyed nodes.
- `MessageNodeComment._ngMessageNode`: back-reference from a transclusion
  comment to its `_messages` key.

## Integration Points

- `$parse`: compiles dynamic message key expressions and truthy `multiple`
  expressions.
- `$attributes`: reads normalized attributes such as `data-ng-messages`,
  `data-ng-message`, `for`, `when`, and `src`.
- `$templateRequest`: fetches included message templates and uses the template
  cache when available.
- `$compile`: compiles included template content into the current messages
  scope.
- Transclusion: creates message elements only when a message is attached.
- Animation service: applies `ng-active` / `ng-inactive` and message
  enter/leave transitions when enabled.

## Edge Cases

- Empty, null, false, numeric zero, empty string, and non-matching collection
  values render no messages.
- String collection values are truthy only when they are non-empty.
- Dynamic message expressions may produce strings, arrays, objects, null, or
  undefined; only matching keys are considered.
- Empty included templates are ignored without throwing.
- Included templates can be overridden by local messages based on DOM position.
- Messages removed before they ever attach still deregister on scope
  destruction.
- A null collection must not leave `_renderLater` stuck in the pending state.

## Destruction And Cleanup

Message instances deregister from `NgMessageCtrl` on containing scope
destruction and when an attached transcluded element is destroyed externally.
Detaching removes the rendered element through animation `leave` when
available, or direct DOM removal otherwise. Included templates stop processing
if their scope is destroyed before the template request resolves.

## Types And Interfaces

`MessageCollection`
: Object map watched by `ngMessages`; truthy values identify active message
keys.

`MessageNodeComment`
: Transclusion comment node carrying the `_ngMessageNode` registration key.

`LinkedMessageCtrl`
: Internal record pairing a `MessageInstance` with its comment anchor.

`MessageInstance`
: Internal attach/detach/test contract implemented by each message directive.

`TruthyExpression`
: Parsed representation of boolean-style `multiple` attributes.

## Testing Notes

- `messages.spec.ts` covers static and dynamic message matching, normalized
  attributes, DOM ordering, active/inactive classes, default messages, included
  templates, multiple mode, and teardown with structural directives.
- Changes to registration or ordering should include tests where template
  inclusion or `ngRepeat` changes registration order.
- Changes to detach behavior should include tests for externally destroyed
  message elements and messages that are removed before first attach.
